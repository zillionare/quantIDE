# Gateway Runtime Follow-ups

## Current Decision

- Gateway 不可连接属于允许场景，不能阻断 pyqmt UI 启动。
- 当前后端已经调整为容忍 gateway 不可用，应用启动后由后台继续重试连接。
- 如果用户最终关闭该功能，系统应退化为仅支持回测和基于历史数据的分析。

## Manual Test Todo

- [ ] 在 gateway 未启动时启动 pyqmt，确认 UI 可以正常进入。
- [ ] 确认 gateway 未连接期间不会重复刷出相同的 `load app_state failed` debug 日志。
- [ ] 确认 gateway 未连接期间，后台会继续重试，但不会影响页面基础导航、回测和历史分析入口。
- [ ] 在 UI 通知实现后，手工验证“继续重试”分支行为。
- [ ] 在 UI 通知实现后，手工验证“关闭实时功能”分支行为，并确认系统退化为 backtest/history-only 能力。

## Implementation Follow-ups

- [ ] 增加初始化完成后的配置入口，允许用户随时修改 gateway 部署地址、端口、prefix 和相关凭证。
- [ ] 在 UI 上增加 gateway 不可用通知。
- [ ] 在通知中提供两个动作：继续重试、关闭实时功能。
- [ ] 关闭实时功能后，在运行时禁用 gateway/livequote 相关能力，并刷新页面可见功能状态。

## User Decisions Needed

- [ ] 初始化后的配置入口放在哪里：独立的“系统设置”页，还是复用 init-wizard 的编辑模式。
- [ ] 通知出现时机：仅首次连接失败时提示，还是运行中断线也提示。
- [ ] “继续重试”是静默后台重试，还是需要允许用户调整重试间隔。
- [ ] “关闭实时功能”是只对当前进程生效，还是持久化写入 `app_state`。
- [ ] 关闭后是否允许用户在 UI 中重新启用，而不重启进程。

## Suggested Repair Directions

1. 先做最小 UI 通知闭环。
   仅在首次确认 gateway 不可用时提示；选择“继续重试”后不再打断用户。

2. 先补配置编辑闭环。
   初始化完成后必须保留一个稳定入口，允许用户迁移 gateway 地址或补填凭证，而不是强依赖重新走初始化向导。

3. 再做功能降级闭环。
   选择“关闭实时功能”后，将运行模式明确切到无实时行情路径，并在 UI 上隐藏或禁用相关入口。

4. 最后决定是否持久化。
   如果要跨重启保留用户选择，需要明确 `app_state` 的字段更新策略和恢复入口。
