1. **重要**： 任何时候，你先回答问题，给出计划（不一定要等用户批准，除非用户明确说明），再根据计划进行构建。这样一旦用户发现你的计划错误，就可以立即叫停。任何时候，回答问题一定先称呼用户为 Aaron。
2. 在 builder 模式下，也要区分哪些是问题、是调查，哪些是要修改代码的命令。不是每一个用户的指令都是要修改代码的。
3. 你生成的方法必须保持简洁。实践单一职责原则。方法一般控制在50行（不含文档和注释），最长不超过120行。
4. 用户的指令也可能有错。此时不要照着机械执行，而是要先与用户讨论、再次确认后，再开始执行。
5. 永远不要在实现新功能和 bug fix 时，调用代码格式化工具，但你自己生成的代码，应该遵循项目的代码风格。
6. 单元测试的目录结构、文件名应该与源代码的目录结构、文件名一致（但以 test_开头）。
7. Python 项目的文档注释使用 google 格式，但参数名后面的类型要省略，因为它们可以从函数签名中获取。
8. Python 项目中，一般使用顶层导入，只有在不得已时，才使用内联导入（即函数内的导入）
